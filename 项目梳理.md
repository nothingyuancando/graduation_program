# 项目梳理

## 1. 项目是做什么的

这个项目是一个基于 `Next.js 16 + React 19 + Supabase + LLM` 的智能学习笔记系统。它不是单纯的“记笔记网页”，而是把“资料导入 -> 内容提取 -> AI 分析 -> 结构化知识 -> 复习练习 -> 学习画像 -> 智能问答”串成了一条完整链路。

从代码实现来看，项目主要解决的是以下几件事：

1. 支持用户创建和管理笔记。
2. 支持从文本、PDF、图片、URL 等多种来源导入内容。
3. 使用大模型自动分析笔记，生成摘要、标签、关键知识点和实体。
4. 基于分析结果生成练习题、学习画像、技能画像。
5. 把笔记和实体组织成知识图谱。
6. 通过 LangGraph + LangChain 搭建一个学习助手 Agent，支持问答、查笔记、分析笔记、出题。

可以把它理解为一个“AI 驱动的个人学习知识库”。

## 2. 技术栈与整体架构

### 2.1 前端

- `Next.js 16`，使用 `App Router`
- `React 19`
- `shadcn/ui + Radix UI`
- `Tailwind CSS 4`

### 2.2 后端能力

- Next.js Route Handlers 作为 API 层
- `Supabase` 作为数据库与后端服务
- `Drizzle ORM` 负责数据结构定义
- `JWT + Cookie` 完成登录态校验

### 2.3 AI 能力

- 自定义 `LLMClient` 统一封装多模型调用
- 支持 OpenAI、DeepSeek、智谱、Kimi、Qwen、SiliconFlow、豆包等提供商
- 使用 `LangChain + LangGraph` 构建学习 Agent

│  ├─ api/                      # 服务端接口

## 3. 项目的核心业务模块

### 3.1 用户认证与登录态

项目通过 Cookie 中的 `auth-token` 保存登录态，后端 API 使用统一方法解析当前用户。

这一层的职责是：

- 登录、注册、退出登录
- 前端全局持有当前用户
- API 路由里校验用户身份

对应核心文件：

- `src/contexts/auth-context.tsx`
- `src/lib/auth.ts`
- `src/lib/jwt.ts`
- `src/app/api/auth/*`

### 3.2 笔记管理

这是整个系统的基础能力。用户可以新增、查看、修改、删除笔记，首页也以笔记列表作为主入口。

笔记不只是普通文本，还带有很多 AI 衍生字段，例如：

- `summary`
- `tags`
- `key_points`
- `themes`
- `entities`
- `mind_map`
- `flashcards`
- `comparisons`

这些结构说明项目的目标不是“保存原文”，而是“围绕原文生成结构化学习资产”。

对应核心文件：

- `src/app/api/notes/route.ts`
- `src/app/api/notes/[id]/route.ts`
- `src/app/page.tsx`
- `src/storage/database/shared/schema.ts`

### 3.3 多模态上传与内容提取

项目支持批量上传文件或 URL。上传后先写入上传会话与处理队列表，再逐个提取文本。

目前代码已经实现的能力包括：

- 文本 / Markdown / HTML 直接读取
- PDF 解析文本
- URL 抓取网页正文
- 图片交给 LLM 做视觉理解与 OCR 式描述

对于音频、视频、压缩包，目前仍然是占位返回，说明项目已经预留了扩展方向，但还没有完成完整解析能力。

对应核心文件：

- `src/app/api/upload/batch/route.ts`
- `src/app/api/upload/sessions/[id]/process/route.ts`
- `src/lib/parsers/pdf.ts`
- `src/lib/parsers/url.ts`
- `src/lib/storage/s3.ts`

### 3.4 AI 笔记分析

这是项目最重要的 AI 处理环节之一。

系统会读取指定笔记内容，并调用大模型生成结构化分析结果，包括：

- 摘要
- 标签
- 实体列表
- 关键知识点

同时，它还会结合用户历史反馈、学习画像、技能画像，让分析更偏向个性化学习场景，而不是仅做通用摘要。

分析完成后会把结果回写到：

- `notes` 表
- `note_entities` 表

并尝试触发相关笔记关系计算。

对应核心文件：

- `src/lib/services/note-analysis.ts`
- `src/app/api/notes/[id]/analyze/route.ts`

### 3.5 测验与练习题生成

系统可以根据笔记内容自动生成练习题，题型固定为：

- 2 道选择题
- 2 道填空题
- 1 道简答题

出题时不仅使用笔记正文，也会参考：

- 标签
- 关键知识点
- 用户薄弱概念
- 技能画像中的难度偏好

这说明测验模块不是独立存在的，而是与“学习画像”和“技能画像”联动。

对应核心文件：

- `src/lib/services/quiz-generation.ts`
- `src/app/api/notes/[id]/quiz/generate/route.ts`
- `src/app/api/quiz/*`

### 3.6 学习画像与技能画像

项目中有两种“用户画像”：

#### 学习画像

更偏“学习表现结果”，主要描述：

- 薄弱概念
- 强项概念
- 兴趣标签
- 学习统计信息

它来自用户已有笔记、知识纠错反馈、复习记录等数据的综合计算。

#### 技能画像

更偏“学习偏好与能力设定”，主要描述：

- 学科水平
- 学习风格
- 学习目标
- 优势领域
- 难度偏好

它会被拼接进 Prompt，影响 AI 的分析和出题策略。

对应核心文件：

- `src/lib/services/learning-profile.ts`
- `src/lib/services/skill-profile.ts`
- `src/app/api/profile/route.ts`
- `src/app/api/skills/route.ts`

### 3.7 知识图谱

知识图谱模块把“笔记”和“实体”都建成节点，再通过关系边组织起来。

目前主要有两类关系：

1. `note -> entity` 的包含关系
2. `note -> note` 的相似关系

相似关系的判定方式比较直接：统计两篇笔记共同包含的实体数，再生成带权重的边。

这部分实现比较清晰，适合后续继续扩展成更完整的知识网络。

对应核心文件：

- `src/app/api/knowledge-graph/route.ts`
- `src/app/api/knowledge-graph/sync/route.ts`
- `src/app/knowledge-graph/page.tsx`

### 3.8 学习 Agent 与聊天能力

聊天模块不是简单地把消息发给模型，而是做成了一个具备工具调用能力的学习 Agent。

这个 Agent 的能力包括：

- 搜索笔记
- 获取笔记详情
- 获取学习画像
- 获取技能画像
- 获取相关笔记
- 分析笔记
- 生成测验

整个执行流程基于 LangGraph：

1. `routerNode` 先根据用户问题做规则分类。
2. 注入对应的系统提示词。
3. 模型决定是否调用工具。
4. 如果需要，就进入工具节点执行。
5. 工具结果返回后，模型继续组织最终答案。

所以项目里的聊天能力，本质上是“基于个人知识库的学习助手”。

对应核心文件：

- `src/app/api/chat/route.ts`
- `src/lib/agent/index.ts`
- `src/lib/agent/graph.ts`
- `src/lib/agent/nodes.ts`
- `src/lib/agent/tools/*`

## 4. 主要业务流程

### 4.1 笔记导入流程

```text
用户上传文件/URL
-> 创建 upload_session
-> 写入 file_processing_queue
-> 按文件类型提取文本
-> 处理结果保存到队列表
-> 后续可整理为笔记
```

### 4.2 笔记分析流程

```text
选择笔记
-> 调用 /api/notes/[id]/analyze
-> analyzeNoteById 读取笔记
-> 拼接用户反馈 + 学习画像 + 技能画像
-> 调用 LLM 输出 JSON
-> 更新 notes 表
-> 重建 note_entities
-> 触发 related notes 计算
```

### 4.3 出题流程

```text
选择笔记
-> 调用 /api/notes/[id]/quiz/generate
-> generateQuizForNote 读取笔记和画像
-> 调用 LLM 生成题目 JSON
-> 写入 quizzes 表
-> 前端进入答题页面
```

### 4.4 聊天 Agent 流程

```text
用户发送消息
-> /api/chat
-> streamLearningAgent
-> buildAgentGraph
-> routerNode 分类问题
-> 模型判断是否调用工具
-> 工具返回结果
-> SSE 流式返回答案
```

### 4.5 知识图谱同步流程

```text
调用 /api/knowledge-graph/sync
-> 读取所有 processed 状态笔记
-> 读取 note_entities
-> 重建 knowledge_nodes
-> 重建 knowledge_edges
-> 前端读取图谱数据进行可视化
```

## 5. 重要数据表

从 `src/storage/database/shared/schema.ts` 可以看出，项目核心数据结构主要有以下几类：

### 5.1 笔记相关

- `notes`：主笔记表
- `note_versions`：笔记版本历史
- `note_entities`：实体提取结果
- `note_relationships`：笔记之间的关系
- `processing_history`：处理流水

### 5.2 上传相关

- `upload_sessions`：批量上传会话
- `file_processing_queue`：文件处理队列

### 5.3 学习分析相关

- `knowledge_feedback`：用户对 AI 结果的纠错反馈
- `user_learning_profiles`：学习画像
- `user_skills`：技能画像
- `quizzes`：练习题集合
- `quiz_attempts`：作答记录
- `study_checkins`：学习打卡

### 5.4 图谱相关

- `knowledge_nodes`
- `knowledge_edges`

这些表说明项目的数据设计是围绕“学习闭环”展开的，而不是只有“笔记内容”本身。

## 6. 最重要的函数梳理

下面是最值得优先理解的一批函数。

### 6.1 `LLMClient.invoke`

文件：`src/lib/llm-provider.ts`

作用：

- 统一调用不同 LLM 提供商
- 接收消息数组
- 发送到兼容 OpenAI Chat Completions 的接口
- 返回统一格式的结果

为什么重要：

- 这是所有 AI 能力的底层入口
- 笔记分析、图片理解、出题等都依赖它

### 6.2 `LLMClient.stream`

文件：`src/lib/llm-provider.ts`

作用：

- 以流式方式调用模型
- 按 chunk 逐步产出文本

为什么重要：

- 适用于聊天、长回答、前端 SSE 输出
- 是后续实时交互体验的基础

### 6.3 `analyzeNoteById`

文件：`src/lib/services/note-analysis.ts`

作用：

- 读取笔记
- 标记笔记状态为 `analyzing`
- 收集用户反馈、学习画像、技能画像
- 调用模型提取摘要、标签、实体、关键点
- 更新 `notes` 和 `note_entities`

为什么重要：

- 它是整个“智能笔记整理”的核心函数
- 决定了项目是否能从原始内容生成结构化知识

### 6.4 `generateQuizForNote`

文件：`src/lib/services/quiz-generation.ts`

作用：

- 读取笔记内容
- 获取用户薄弱概念和技能偏好
- 让模型生成固定结构的题目 JSON
- 将题目写入 `quizzes`

为什么重要：

- 这是项目把“笔记”转成“练习”的关键函数
- 直接体现学习系统的闭环能力

### 6.5 `computeLearningProfile`

文件：`src/lib/services/learning-profile.ts`

作用：

- 统计用户笔记标签，推断兴趣
- 从知识点置信度中提取强弱项
- 从纠错反馈中修正强弱项
- 从复习记录中补充概念掌握程度
- 输出学习画像结果

为什么重要：

- 它把“静态笔记库”变成了“可分析的学习数据”
- 后续推荐、出题、解释风格都可以基于它扩展

### 6.6 `getOrCreateLearningProfile`

文件：`src/lib/services/learning-profile.ts`

作用：

- 先尝试读取已有画像
- 如果没有，或要求强制重算，就调用 `computeLearningProfile`
- 再把结果写回数据库

为什么重要：

- 它是学习画像的统一入口
- 页面与 API 不需要关心画像是否已存在

### 6.7 `getOrCreateSkillProfile`

文件：`src/lib/services/skill-profile.ts`

作用：

- 读取用户技能画像
- 如果没有数据，返回默认画像结构

为什么重要：

- 保证个性化 Prompt 始终有稳定数据结构
- 减少调用侧的判空复杂度

### 6.8 `updateSkillProfile`

文件：`src/lib/services/skill-profile.ts`

作用：

- 更新或创建用户技能画像
- 支持学科水平、学习风格、目标、偏好等字段

为什么重要：

- 是用户主动调节 AI 行为的重要入口

### 6.9 `formatSkillContextForPrompt`

文件：`src/lib/services/skill-profile.ts`

作用：

- 把技能画像转成适合 Prompt 的自然语言上下文

为什么重要：

- 它连接了“结构化用户画像”和“LLM 输入”
- 是个性化效果的关键桥梁

### 6.10 `processTextFile`

文件：`src/app/api/upload/sessions/[id]/process/route.ts`

作用：

- 统一处理文本类文件与 URL
- PDF 走 `parsePdf`
- URL 走 `fetchUrlContent`
- 纯文本直接解码

为什么重要：

- 它是多来源内容进入系统的统一入口之一

### 6.11 `processImageFile`

文件：`src/app/api/upload/sessions/[id]/process/route.ts`

作用：

- 给图片生成可访问 URL
- 把图片交给多模态模型分析
- 返回图像内容说明或提取文本

为什么重要：

- 让系统具备“图片转可学习文本”的能力

### 6.12 `parsePdf`

文件：`src/lib/parsers/pdf.ts`

作用：

- 解析 PDF Buffer，提取纯文本

为什么重要：

- 是上传场景里最基础也最常用的解析函数

### 6.13 `fetchUrlContent`

文件：`src/lib/parsers/url.ts`

作用：

- 抓取网页 HTML
- 去掉脚本、样式、导航等噪音
- 提取标题与正文文本

为什么重要：

- 让网页资料也能作为学习输入源

### 6.14 `buildAgentGraph`

文件：`src/lib/agent/graph.ts`

作用：

- 构建 LangGraph 状态图
- 串联路由节点、模型节点、工具节点

为什么重要：

- 它定义了智能问答的执行框架
- 决定 Agent 如何“思考”和“调用工具”

### 6.15 `routerNode`

文件：`src/lib/agent/nodes.ts`

作用：

- 根据用户提问快速判断意图类型
- 注入对应系统提示词
- 拼入技能画像上下文

为什么重要：

- 提升 Agent 响应的针对性
- 避免每次都完全依赖模型自己理解意图

### 6.16 `shouldContinue`

文件：`src/lib/agent/nodes.ts`

作用：

- 判断当前是否还需要继续执行工具调用
- 限制最大迭代次数

为什么重要：

- 防止 Agent 无限循环调用工具

### 6.17 `streamLearningAgent`

文件：`src/lib/agent/index.ts`

作用：

- 加载历史对话
- 运行 Agent 图
- 逐步输出 token / tool 事件
- 最终保存对话历史

为什么重要：

- 是聊天接口真正调用的主入口

### 6.18 `searchNotesTool`

文件：`src/lib/agent/tools/search-notes.ts`

作用：

- 从用户问题里抽取关键词
- 为笔记打分排序
- 返回最相关笔记

为什么重要：

- 是 Agent 连接个人知识库的最常用工具

## 7. 这个项目当前已经做到了什么

从现有代码判断，项目已经具备一个毕业设计级别的完整雏形，且不只是 Demo 页面，而是已经实现了较完整的数据流：

- 有登录注册
- 有笔记 CRUD
- 有批量上传与文件处理队列
- 有 PDF / URL / 图片内容提取
- 有 AI 分析与实体抽取
- 有个性化学习画像与技能画像
- 有练习题生成
- 有知识图谱同步
- 有具备工具调用能力的聊天 Agent

也就是说，这个项目的核心不是“调用了一次大模型 API”，而是已经形成了一个相对完整的“学习增强系统”。

## 8. 当前实现里的亮点

### 8.1 模块拆分比较清晰

AI 服务、上传处理、画像服务、Agent 工具、API 路由分层都比较明确，后续维护性不错。

### 8.2 数据设计比较完整

不仅存储原始笔记，还存储实体、关系、反馈、练习、画像、打卡数据，说明项目思路是系统化的。

### 8.3 个性化能力已经落到代码

学习画像和技能画像不是摆设，已经真正参与笔记分析和出题 Prompt。

### 8.4 Agent 不是空壳

聊天模块已经能调用实际工具，不只是简单对话接口。

## 9. 当前还可以继续加强的地方

如果后面要继续完善，这几个方向最值得做：

1. 补齐音频、视频、Office 文档的真实解析能力。
2. 为知识图谱增加增量同步，而不是每次全量清空重建。
3. 给笔记分析结果增加更严格的 JSON 校验与兜底逻辑。
4. 增加更多复习策略，比如间隔重复、错题本、推荐复习。
5. 进一步把 `mind_map`、`flashcards`、`comparisons` 等字段真正用起来。
6. 为 Agent 增加更多工具，例如导出总结、推荐复习计划、对比两篇笔记等。

## 10. 一句话总结

这个项目本质上是一个“面向学习场景的 AI 知识管理平台”。它以笔记为核心，把多模态导入、内容理解、知识结构化、测验生成、学习画像、知识图谱和智能问答整合到了一起，已经具备比较完整的产品原型和毕业设计展示价值。
